# 需求文档 —— 程序结构依赖图

## 1. 背景与动机

sc 的立身之本是「**程序即结构**」：一个程序不是一堆文本，而是一棵由结构对象组成的
层级树 —— 顶层是 `def`（类型）、`fnc`（函数）、`var`/`let`（变量/常量），以及它们的
变体 `rpc`/`tls`、方法、维度等。

结构对象之间通过**相互调用与引用**产生依赖，这些依赖天然构成一张**有向图**。
这与 sc 运行时的 `tok` 分布式依赖机制同构，但二者所处层面不同：

| | 层面 | 图的语义 |
|---|---|---|
| `tok` / `dep` / `map` | **运行期** | token 值在计算图上按序传播、激活 |
| 结构依赖图（本项目） | **编译期** | 结构对象间的符号引用关系、静态可达 |

> **术语独立**：本项目自成一套词汇（node/edge/active/root/…），与 `tok` 运行时机制
> **无继承关系**，避免概念混淆。

我们希望编译器像烘焙 `tok` 图那样，为**每个程序结构对象**烘焙出它的依赖关系，
从而回答：谁依赖谁？从入口出发哪些对象是「活的」？程序真实的结构骨架长什么样？

## 2. 用户诉求（原始 5 点）

1. 程序即结构，是树层级结构。
2. 程序的互相调用构成依赖图，和 `tok` 机制类似（同构，非复用）。
3. 编译器为每个程序结构对象烘焙出图依赖关系。
4. 可执行文件的依赖起点是 `main` 入口；被 `main` 递归依赖的对象视为**激活**。
5. 依赖图可导出（如 JSON），并形成可视化（HTML）。

## 3. 已定型的设计取舍（决策记录）

以下取舍已与需求方确认，作为实现约束：

1. **粒度（分两层，一期只做上层）**
   - 最终目标两层：**模块间**（导出函数/对象之间的跨模块依赖）+ **模块内部**（单元内部依赖）。
   - **一期（P0–P3）统一为 Decl 级**：节点 = 顶级关键字声明对象（`def`/`fnc`/`var`/`let`/`tls`/方法…）。
   - 模块间 vs 模块内的分层视图作为后续增强（在同一份 Decl 级图上按 `module` 折叠即可）。

2. **默认范围 = 整程序**
   - `--graph` 默认递归解析全部 `inc` 依赖，union 成整程序图（激活语义所需）。
   - `--graph=unit` 退化为**仅当前编译单元**（含 `add` 内联），外部符号作为叶子节点。

3. **边种类一次到位**
   - 首个可用版本即覆盖全部边种类：`call` / `type` / `read` / `write` / `method` /
     `construct` / `macro` / `tokdep` / `import`（详见 design.md §3）。

4. **激活根**
   - 可执行单元：根 = `main`。
   - 库单元（无 `main`）：根 = 全部 `@导出` 对象 —— 相当于「多个顶级入口」。

5. **可视化 = HTML 直出为默认，兼可输出 JSON**
   - `scc app.sc --graph -o g.html`：编译器直出**自包含 HTML**（数据内嵌 + 查看器，离线可用）。
   - `scc app.sc --graph -o g.json` / `--graph`（stdout）：输出**原始 JSON**，供用户自定义扩展/显示。
   - HTML 是默认交付形态；JSON 是可编程的数据契约。

6. **术语独立于 tok**
   - 不复用 `tok`/`dep`/`map`/`form`/`back` 词汇；本项目用 node/edge/root/active/reach 等中性术语。

## 4. 非目标（本项目暂不做）

- 不追求指针/动态分派的**完备**解析（动态点保守记「可能边」，不保证无漏无误）。
- 不做语句/表达式级的细粒度图（一期止于 Decl 级）。
- 不改动 codegen / 运行时行为：纯**只读分析 + 导出**，不影响编译产物。
- 不做增量/持久化缓存（后续可选）。

## 5. 验收总纲

- 新增 `--graph` 模式，行为与 `--ast`/`--api` 对齐（模式派发、`-o` 后缀推导）。
- 对示例工程（如 `templates/demo/*`）能导出结构正确的 JSON，且回归可锁定。
- HTML 可离线打开，展示节点/边、激活高亮、按模块/边种类过滤。
- 全量回归（`./tests/run.sh`）保持 0 fail。
