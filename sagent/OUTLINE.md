
# sc agent —— 第三大版本（最终目标版本）：LLMs 的原生开发语言

> 三个版本的弧线：v0.9 语言与编译器基座 → v0.9.1 并行计算体系 + `.scenv` 域基
> → **v1.0 agent：sc 的初衷**。前两个版本为本篇备齐了"传感器"（编译/测试/图导出
> 的确定性反馈）与"域基范式"（目录即配置、文件即状态）。

全文分三层：**公理层**（§0，一切设计的推导起点）→ **机制层**（§1–§5，
工具形态与运行机制）→ **体系层**（§6–§10，四大认知体系与路线）。

---

# 篇一·公理层

## 0. 公理与推论

**公理（LLM 本质）**：一个线性串行的信息序列形成的因果导向。

由此推出本设计的全部取向：

1. **上下文质量 = 因果链的信噪比**。会话历史累积会让失败尝试、无关细节稀释
   因果链——所以上下文基于 loop **每次动态构造**，而非历史会话。
2. **生成是不确定的，验证提供确定性**。LLM 产出的价值下限由验证体系托底——
   而 sc 生态的编译器/测试器/图导出器天然是机器可验证的反馈源（§7）。
3. **状态必须外置、可见、可版本化**。所有上下文信息以文件方式保存在项目
   `.sagent/` 下——上下文本身成为工程制品：可 git、可 diff、可人工修剪、
   可脚本生成。
4. **原子工具 + 文件 + 脚本 = unix 哲学**。sca 只提供原子操作，不做重度集成
   耦合；通过手动编辑 `.sagent/` 的文件与结构、配合 sh/ps1 脚本，即可整合出
   coding / plan / data-analyse / … 各类 agent——不同 agent 只是**不同的目录
   模板和脚本**，不是不同的软件（与 `.scenv` 同一设计基因）。

# 篇二·机制层

## 1. 工具形态

类似 git 的命令行辅助工具 **sca**：

- **一期独立二进制**；二期视应用效果决定是否集成进 scc（子命令形态）；
- 所有上下文信息全部以文件方式保存在项目目录下的 `.sagent/`；
- 以 **loop 为执行单元**：一次 sca 调用，执行一次 loop；
- 每个 loop 的初始上下文动态构造（§5），而非历史会话；
- 工具功能原子化，编排交给脚本（公理 4）。

```sh
# 初始化，生成 .sagent 目录结构
sca init

# 基本会话（单次 loop）
sca "hi, llm" [opts]
```

**工具调用面**：scc + git + curl + cmdline——

| 角色 | 工具 | 职责 |
|---|---|---|
| 传感器/执行器 | scc | 编译、测试、`--api`/`--graph`/`--ast` 结构导出、运行验证 |
| 时间轴/回滚 | git | 一 loop 一 commit，历史即审计，失败 loop 可 revert |
| LLM 通道 | curl | 对接 LLM API（最小集：OpenAI 兼容协议）【待定：本地模型】 |
| 万能手 | cmdline | 受控的 shell 执行（白名单/确认策略，§4 config） |

## 2. loop 单元

**要素**：目的，目标，差异，验证。

| 要素 | 含义 | 生存期 |
|---|---|---|
| 目的 | why——task 级不变量，为何做这件事 | 整个 task 不变 |
| 目标 | what——本 loop 结束时可验收的状态 | 单 loop |
| 差异 | 现状与目标之间的 diff（分析体系的输入，§8） | 每 loop 重新测得 |
| 验证 | 测得差异的机器手段（编译/测试/对拍/结构 diff） | 随目标定义 |

**生命周期**（一次 sca 调用）：

```
构造初始上下文（§5）→ 调 LLM → 执行工具调用 → 验证（§7）
  → 复盘写回（陷阱/事实 → .sagent）→ git commit（代码 + .sagent 同一提交）
```

**复盘**：陷阱，事实——loop 结束时由 LLM 自我复盘写回：
- **陷阱**：本 loop 踩过的坑（失败的路径及原因），供后续 loop 避让；
- **事实**：本 loop 确认的新事实（环境、行为、约定），沉淀进决策体系（§9）。

**与 git 的契约**：一 loop 一 commit。`.sagent/` 变更与代码变更进同一提交——
上下文与代码的因果对应被 git 历史固化；失败 loop revert 即整体回滚。

## 3. task 会话

**本质：loop 集合 + 上下文闭包**。

- **loop 集合**：task 由一组 loop 单元构成，对目标完成最终验证、对过程进行
  推导分析；
- **上下文闭包**：task 圈定了其内所有 loop 构造上下文的选材边界——
  goal/plan/state/memory 这些绑定活在 task 环境里，**loop 是在闭包内求值的
  过程**；闭包之外的信息不进入上下文（公理 1 的信噪比保证），闭包之内的绑定
  随 loop 演进被更新（state 追加、plan 消费、memory 沉淀）；
- 编排驱动：一期由 sh/ps1 脚本循环调用 sca（公理 4），sca 不内置编排；
- 终止条件：最终验证通过 | loop 预算耗尽 | 人工中断；
- **过程推导**：task 结束时扫描全部 loop 复盘，把陷阱/事实沉淀进决策体系
  （§9），task 归档 = 闭包封存（可整体回放）。

## 4. `.sagent/` 目录规范（草案，随实现校准）

```
.sagent/
  config.sc            # LLM 接入（endpoint/model/密钥引用）、loop 预算、工具白名单
  prompts/             # 协议 prompt（可手编，loop 自动加载；loop.md/review.md）
  task/                # 当前 task
    goal.md            # 目的 + 最终验证标准
    plan.md            # 待发生：计划（loop 目标队列，可由 LLM/人共同编辑）
    state.md           # 事实：已发生序列（每 loop 追加一行摘要）
    loop-NNN/          # 每 loop 档案
      context.md       # 本 loop 构造的初始上下文快照（审计/复现用）
      actions.jsonl    # 工具调用序列（可回放）
      review.md        # 复盘：陷阱/事实
  memory/              # 决策体系四分类（§9）
    structure.md       # 结构和关系：空间/概念/意义/理解（模块、依赖、导向）
    paths.md           # 条件和发生：实在/域/范围（路径、分支）
    facts.md           # 前提和习惯：约束/风格/优化/事实（环境、默认、约定）
    history.md         # 事实和计划：发生序列/存在位置/未来方向/事务排期
  archive/             # 已完成 task 归档（task 目录整体移入）
```

- `config.sc` 复用 sc 配置文件语法（同 `.scenv/env.sc` 范式）；密钥不入库，
  存环境变量引用；
- 目录整体 git 入库（除密钥）；手动编辑任何文件都是合法操作——这是"原子工具
  + 文件整合"的落点。

## 5. 上下文构造机制

每个 loop 的初始上下文 = 动态选材拼装，选材来源（按优先序）：

1. **task 三件**：goal.md（目的）+ plan.md 中本 loop 的目标 + state.md 尾部
   （最近已发生）；
2. **记忆四件**：memory/ 按目标相关性选取（一期：全量小文件直入；
   二期：结构化检索）【待定：选材算法】；
3. **上一 loop 复盘**：review.md（陷阱直接进入上下文避免重蹈）；
4. **现场探测**：即时执行 `scc --api`/`--graph`、`git status/diff` 等注入
   实时事实——**这是 sc 作为原生语言的独占优势：现场事实由编译器保证正确**。

构造器可插拔：sca 提供默认构造器；`.sagent/hooks/` 下的脚本可覆盖/追加
【待定：hook 接口】。

# 篇三·体系层

## 6. 解构体系

**本质：消除歧义**。与验证、分析、决策并列的第四大体系，且位于因果链最前端：
生成的失败大多不是能力问题而是歧义问题——目标含混 → 生成漂移 → 验证无标准
可判。解构把模糊需求投影为无歧义的目标空间，是验证得以成立的前提。

四个动作（每个针对一类歧义）：

| 动作 | 消除的歧义 | 落点 |
|---|---|---|
| 概念对齐 | 语义歧义（同词异义/异词同义）——映射到决策体系的语义空间（§9 结构和关系） | goal.md 术语定义 |
| 边界划定 | 范围歧义（做到哪为止）——域/范围（§9 条件和发生） | task 闭包边界（§3） |
| 标准具化 | 判定歧义（怎样算完成）——机器可判定的验收标准 | goal.md/plan.md，验证（§7）准入 |
| 未知显式化 | 假设歧义（没说的当成什么）——隐含假设翻成显式待定项 | plan.md 待探测/【待定】 |

**时机**：
1. task 建立时——goal/plan 的产生过程本身就是一次完整解构；
2. 每 loop 目标入队前——plan.md 队列的准入检查（标准不可判定则退回解构）；
3. 差异归因后——分析（§8）发现差异源于歧义（而非执行）时，回到解构重述目标。

**四体系闭环**：

```
解构（消除歧义，事前定义）→ 验证（判定是非，事后测量）
  → 分析（差异回溯，归因修订）→ 决策（世界观与历史，提供依据）
  → 新知沉淀回世界观，供下次解构对齐
```

## 7. 验证体系

**本质：目标与标准——一个二元判定是非的过程**。这是 agent 的核心：
为生成提供确定性。

- 判定为**是**：目标达成，loop/task 收敛；
- 判定为**非**：则必须**产出差异**，作为分析体系（§8）的输入——验证不能只说
  "错了"，必须说出"差多少、差在哪"。

分层（每层 = 一个二元判定 + 判非时的差异产物）：

| 层 | 判定手段 | 差异产物（判非时） |
|---|---|---|
| 语法/语义 | scc 编译 | 错误定位（文件:行 + 原因） |
| 行为 | `--test` 单测、golden 快照、`--check=*` | 失败用例 + 期望/实际 diff |
| 数值 | demo 对拍（allclose 范式，同 spc） | 误差分布（哪个张量、多大偏差） |
| 结构 | `--api` 接口 diff、`--graph` 依赖图 diff | 接口/依赖的增删改集合 |
| 过程 | actions.jsonl 回放审计 | 与计划动作序列的偏离点 |

目标与标准写在 goal.md / plan.md（§4）——**没有事先写下的标准，就没有合法的
验证**；标准的可判定性是 loop 目标的准入条件，而达成这个准入正是解构体系
（§6）的职责。

## 8. 分析体系

**本质：差异和回溯**。这个过程与机器学习在本质上是同一的：

| 分析 | 机器学习 |
|---|---|
| 差异 | 求导（损失对输出的导数） |
| 回溯 | 反向传播 |
| 节点上每个可能的置信度 | 梯度 |
| 修订最不确定的节点 | 参数更新 |

运作：验证产出差异后，沿因果链**逐节点回溯**（本 loop 的动作 → 动作的决策
依据 → 决策的前提假设）；**每回溯一个节点，就是在那个时点寻找差异的合理性**
——此刻的每一个可能都具有一个置信度（即梯度），**差异就来自最不确定的方面**，
那里就是优先修订点。

```
差异（求导）→ 回溯（反向传播：动作 → 决策 → 前提，逐点评估置信度）
  → 定位置信度最低的节点 → 修订（plan.md 的目标 / memory 的前提）
  → 下一 loop（前向传播）
```

差异为零 = 收敛；差异不收敛 N 次 = 触发升级（换路径 / 求助人工 /
记入陷阱后放弃该路径）【待定：N 与升级策略】。若归因发现差异源于歧义
（目标本身含混），则退回解构体系（§6）重述目标。

## 9. 决策体系

**本质：历史和世界观**。决策 = 用世界观解释历史、推演未来。四分类即世界观
与历史的结构化——本质是四种不同读写模式的数据：

| 分类 | 内涵 | 角色 | 数据形态 | 读写纪律 |
|---|---|---|---|---|
| 结构和关系 | 空间、概念、意义、理解 | 世界观：语义空间（事物如何关联、意味着什么） | 图 | 由工具再生（`scc --graph/--api`），人工只注释 |
| 条件和发生 | 实在、域、范围 | 世界观：什么在什么域/范围内成立与发生 | 决策树 / 守卫条件 | loop 复盘追加分支结论 |
| 前提和习惯 | 约束、风格、优化、事实 | 先验：决策的默认取向与硬边界 | 常量表 | 只追加/修正，事实过期须删除 |
| 事实和计划 | 发生序列、存在位置、未来方向、事务排期 | 历史与未来：时间轴 | 日志 + 队列 | state.md 追加 / plan.md 队列消费 |

三层合成：前两行构成**世界观**（静态模型 = 语义空间 + 条件成立域），第三行
是**先验**（默认与边界），第四行是**历史与排期**（时间轴上的位置与方向）。
**决策过程 = 在世界观之上、以先验为约束、从历史所处位置推演未来方向**——
这也正是 §8 回溯时逐节点置信度的来源：某节点的决策若在世界观/先验里有强依据，
置信度高；若是无依据的临场猜测，置信度低，差异优先归因于它；同时它也是
解构（§6）概念对齐的靶子：术语必须落在语义空间已有的概念上，否则先扩充世界观。

纪律要点：**结构文件不靠 LLM 手写维护**——凡工具可再生的（依赖图、接口摘要）
一律再生，LLM 只消费；记忆文件保持小而准（同 AGENTS.md 的"诚实标注"原则）。

## 10. 分期路线

- **M1 最小可用 loop**：`sca init` + 单次 loop 全生命周期（构造上下文 →
  curl 调 LLM → 应用文件修改 → scc 验证 → 复盘写回 → git commit）。
  验收：在 sc 仓库内完成一个真实小修改任务（如加一个 builtin 函数 + 测试）。
- **M2 task 编排**：plan.md 队列消费、终止条件、task 归档、示例编排脚本
  （coding agent 模板）。验收：多 loop 完成一个中等任务且历史可审计回放。
- **M3 决策记忆**：memory/ 四分类读写纪律落地、结构文件自动再生、
  上下文选材优化。验收：跨 task 的陷阱复用可观测（同类错误不再犯）。
- **M4（评估后定）**：集成进 scc（子命令）；本地模型通道；多 agent 模板库
  （plan / data-analyse / …）。
